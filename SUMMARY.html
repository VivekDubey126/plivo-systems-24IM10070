The Flaky Network: Architecture Summary

1. Overview

The system is built on a custom Selective Repeat ARQ (Automatic Repeat reQuest) protocol over UDP. Given the strict bandwidth overhead constraint (≤ 2.0x), aggressive Forward Error Correction (FEC) (such as packing redundancy into every packet) was mathematically infeasible. Instead, the architecture focuses on a lightweight NACK-based feedback loop that requests retransmissions only when packets are definitively dropped.

2. Sender Architecture

The sender acts as a fast-forwarder with a memory buffer. It maintains a ring buffer (history[65536]) holding previously sent payloads. The sender multiplexes using select() to monitor two UDP sockets non-blockingly:

  • Source Socket: Forwards incoming 164-byte packets (4-byte sequence + 160-byte payload) immediately to the relay network, while storing a copy in the history buffer.
  • Feedback Socket: Listens for 4-byte NACKs containing sequence numbers. Upon receipt, the sender instantly retrieves the corresponding payload from the history buffer and retransmits it.

3. Receiver Architecture

The receiver's primary goal is to minimize playout delay. It does this by forwarding valid packets to the player immediately rather than attempting to enforce a strict playout clock. It uses a boolean array to track which sequence numbers have been received and tracks the highest_seq seen so far.

When a packet arrives:

  1. The receiver checks if the sequence number is new. If it is, the packet is instantly sent to the player.
  2. The receiver updates its highest_seq.
  3. The receiver identifies any sequence gaps (missing frames between the old and new highest_seq) and immediately dispatches a 4-byte NACK to the feedback path.

Periodic Re-NACKing: To ensure reliability against lost NACKs (since the feedback path also traverses the flaky network), the receiver utilizes a timed select loop (10ms timeout). Every 10ms, it scans the recent sequence space. If a packet has been missing for over 40ms (an estimated safe RTT), a duplicate NACK is dispatched.

4. Design Choices & Tradeoffs

  • No FEC: Avoided FEC completely to easily pass the 2.0x bandwidth budget (our overhead sits around 1.10x). The tradeoff is a reliance on the network's Round Trip Time (RTT), necessitating a slightly higher but still generous delay_ms of 200.
  • Raw 4-byte NACKs: Minimized feedback bandwidth by sending only the 4-byte sequence number as the payload for NACKs.
  • Cross-Platform Polling: Switched to robust select() loops to ensure 100% compatibility across POSIX (Linux/macOS/WSL) and Windows (Winsock2) environments.
