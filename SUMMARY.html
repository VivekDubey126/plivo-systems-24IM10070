The Flaky Network — Architecture Summary

Hybrid FEC + NACK Zero-Allocation Hot Path Circular Jitter Buffer C++17

1. Problem Statement

The system must reliably deliver 160-byte audio/video frames, generated every 20 ms, across a relay network that introduces packet loss, variable delay (jitter), reordering, and duplication. The hard constraints are a deadline-miss rate of ≤ 1.0% and a bandwidth overhead of ≤ 2.0× the raw stream. The competitive metric is the lowest achievable delay_ms.

2. Why Pure ARQ (NACK-Only) Is Insufficient

A NACK-only scheme requires detecting a loss, sending a NACK, relaying it back to the sender, and waiting for a retransmission — a minimum of one full network round-trip time (RTT). With a relay introducing up to 40–80 ms of one-way delay, the ARQ RTT alone could exceed 100 ms, making any delay_ms below ~150 ms physically impossible under pure ARQ.

3. The Hybrid FEC + NACK Architecture

3.1 Sender — FEC Piggybacking

For every frame i received from the harness (on port 47010), the sender bundles the current payload and the previous payload into a single 328-byte UDP datagram sent to the relay uplink (47001):

BytesContent
0–3Current sequence number (big-endian uint32)
4–163Current 160-byte payload
164–167Previous sequence number (big-endian uint32)
168–327Previous 160-byte payload

This doubles the raw payload bytes, consuming exactly the 2.0× budget. A 1024-slot power-of-two ring buffer (indexed by seq & RING_MASK) stores recent payloads for instant O(1) retrieval on a NACK.

3.2 Receiver — Zero-Allocation Circular Jitter Buffer

The receiver maintains a statically allocated RxSlot jb[1024] array. Each slot holds a 160-byte payload, a boolean received flag, and a NACK timestamp. Indexing is jb[seq & BUF_MASK] — a single bitwise AND, giving O(1) access with no dynamic memory allocation whatsoever in the receive hot path. This eliminates GC pauses and heap fragmentation latency spikes entirely.

3.3 FEC Instant Recovery

When a 328-byte packet arrives for sequence N, it also carries N-1. If packet N-1 was dropped by the relay, the receiver recovers it immediately from this piggybacked copy — zero extra round-trips, zero added delay. This handles all single-packet loss events (the dominant case on typical profiles).

3.4 NACK Fallback for Burst Loss

If a gap of ≥2 consecutive packets is detected (FEC cannot recover both), the receiver immediately sends a 4-byte NACK (the missing sequence number, big-endian) to port 47003. The sender listens on port 47004 and retransmits the requested frame from its ring buffer within microseconds. NACKs are retried every 30 ms if no recovery arrives.

3.5 In-Order Drain

A next_deliver counter tracks the next expected sequence. After every receive event the receiver drains contiguous received slots, sending each as a strict [4-byte seq][160-byte payload] = 164-byte datagram to the player on port 47020.
